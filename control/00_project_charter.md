# 00_project_charter.md

## 1. 项目身份与总目标

**项目名称（工作名）**  
CtR–CtB 协调组合优化：以 CtR 为主目标、以 CtB 为结构约束的 long-only 大类资产 ETF 组合研究。

**项目性质**  
本项目不是“论文写作”和“代码实现”两件彼此分离的任务，而是一条统一研究管线：

> 笔记与理论来源 → 总控文件 → 数学对象 → 模型契约 → 求解器契约 → 数据契约 → 实验注册 → 结果导出 → 图表/表格 → 论文正文 → 答辩材料

**总目标**  
在统一符号、统一定义、统一结果口径的前提下，完成一套可复现、可写成硕士论文、可生成答辩材料的研究系统。核心研究主题为：

1. CtR（Contribution to Risk）的理论与 long-only 可操作化；
2. CtB（Correlation to Basket）的理论、与 CtR 的联系，以及 long-only 下的可操作化；
3. 以 CtR 为主目标、CtB 为结构约束的协调模型；
4. 对应求解器、回测框架、结果体系与论文表达。

## 2. 一级理论来源与优先级

### 2.1 一级理论来源（最高优先级）

本项目的一级理论来源是用户上传笔记中的数学对象、定义、推导与问题设置，尤其包括：

- 金融市场、收益率、协方差矩阵、Markowitz/Sharpe 主线；
- CtR/ERC 的定义与等风险贡献思路；
- CtB（Correlation to Basket）及其与 CtR 的关系；
- 凸性、强凸性、最优性条件；
- “先梯度、后牛顿”的算法路线；
- long-only / no-short 约束语境。

### 2.2 二级来源

二级来源为外部文献、现有公开代码、历史对话方案、参考仓库与常见实践。  
二级来源只能用于：

- 补充文献定位；
- 对照已有方法；
- 借鉴实验与图表呈现；
- 补足工程实现细节。

### 2.3 来源冲突时的处理原则

若一级来源与二级来源冲突，**优先服从一级来源**。  
若一级来源内部存在记号不一致、公式表达不一致或建模口径不一致，必须：

1. 显式报告冲突；
2. 提出统一方案；
3. 在 `01_notation_master.md` 与 `02_definition_formula_ledger.md` 中完成统一；
4. 未经说明，不得在后续章节、代码或图表中自行漂移。

## 3. 研究边界

### 3.1 做什么

本项目聚焦于以下边界内的问题：

- 研究对象为 **大类资产 ETF**；
- 组合主要考虑 **long-only** 语境；
- CtR 为主目标；
- CtB 作为相关结构约束或容忍带；
- 研究输出包括理论推导、数值求解、回测实验、图表表格、论文正文、答辩材料。

### 3.2 不做什么

本项目原则上**不把以下方向作为主线**：

- 高频交易、日内择时、预测驱动 alpha 模型；
- 个股超大资产池优化；
- 复杂机器学习预测模型作为论文核心；
- 以收益预测为主的主动管理论文；
- 将问题泛化为“所有风险模型都做一点”的宽泛综述。

### 3.3 long-only 约束的重要性

long-only 不是附属条件，而是本项目叙事的核心结构条件。  
本项目将重点讨论：

- 在 long-only 约束下，CtR 的可操作化；
- 在 long-only 约束下，CtB 的定义、性质与数值实现；
- 为什么主模型采用 “CtR 主目标 + CtB 约束” 比直接做抽象“双平价主目标”更稳健、更可答辩。

## 4. 核心研究问题

本项目围绕以下核心研究问题组织：

### RQ1. 理论对象问题
如何在统一的投资组合数学框架下同时刻画 CtR 与 CtB？

### RQ2. 可操作化问题
在 long-only 约束下，CtR 与 CtB 各自应如何定义、标准化与数值求解？

### RQ3. 关系问题
CtR 与 CtB 在何种条件下存在比例、退化、对齐或偏离关系？

### RQ4. 建模问题
为什么主模型选择 “CtR 主目标 + CtB 约束/容忍带”，而不是将二者简单并列为同等主目标？

### RQ5. 数值实现问题
如何设计统一求解器，使其能够覆盖 CtR-only、CtB-only 与主模型三类问题，并体现“先梯度、后牛顿”的路线？

### RQ6. 实证问题
在大类资产 ETF 组合中，引入 CtB 结构约束是否能够改变风险配置、相关结构、换手、稳健性与解释性？

## 5. 论文主张与输出原则

### 5.1 项目最终输出

本项目最终必须同时产出两类结果：

#### A. 面向论文与答辩的结果
- 可直接入论文的图表；
- 可直接入论文的 LaTeX/Markdown 表格；
- 章节正文；
- 附录推导；
- PPT 与讲稿。

#### B. 面向复现与后续分析的结果
- CSV/JSON 机器可读结果；
- 求解器诊断文件；
- 实验注册与参数记录；
- 可供后续 GPT 读取与分析的结构化结果包。

### 5.2 统一口径原则

所有输出必须满足：

- 同一对象只使用一套记号；
- 同一指标只使用一个主名称；
- 同一图表必须有底层数据；
- 同一 run 必须能追踪对应参数、数据窗口、协方差估计器、策略名称与导出文件。

## 6. 文件治理原则

### 6.1 三层冻结机制

本项目所有文件按三层管理：

#### F1：硬冻结（Hard Freeze）
- 一经确认，原则上不改；
- 仅在发现理论冲突、定义错误或顶层结构性失误时，经过显式“解冻—修订—再冻结”流程修改。

#### F2：理论稳定后冻结（Theory-Stable Freeze）
- 在核心理论与模型稳定后冻结；
- 允许在研究推进早期进行一次或少数几次结构性更新。

#### F3：滚动更新（Rolling）
- 与实验、结果、写作、答辩同步更新；
- 直到项目末期才最终冻结。

## 7. 与 GPT 协作的强约束

后续所有项目对话默认遵守以下规则：

1. 每次回答都先检查哪些文件受到了新信息影响；
2. 自动识别需要更新的 F2/F3 文件；
3. 若本次回答导致文件内容变化，则给出更新后的版本；
4. 若内容触及 F1 文件，则必须显式说明：
   - 为什么需要解冻；
   - 改动范围；
   - 改动后的再冻结说明；
5. 未经说明，不得擅自改动：
   - 统一符号；
   - 顶层研究问题；
   - 章节主线；
   - 核心模型角色分工；
   - 指标口径。

## 8. 结果评价标准

本项目评价标准按以下顺序排序：

1. **理论一致性**：符号、定义、公式与主张不漂移；
2. **论文可写性**：能自然转化为章节、图表、表格与结论；
3. **代码可复现性**：同一配置可复现实验结果；
4. **答辩可解释性**：每个设计选择都能被说明；
5. **结果稳健性**：参数选择、样本切分、诊断输出合理；
6. **图表质量**：图表服务机制解释而非装饰。
